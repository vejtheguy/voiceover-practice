Using Hidden Headings inside Navigation Elements

Visually Hidden CSS Techniques

When wrapping a list of links inside a native nav element without a heading, I noticed that the screen reader tends to over-announce structural information. Because there is no explicit label for the navigation block, the screen reader will often combine the landmark, list, and first link data into a single, long, repetitive announcement.

My Implementation:

To fix this, I found that we can add a visually hidden h2 element at the beginning of the nav block and link it to the container using an id and the aria-labelledby attribute. Adding this heading allows the screen reader to announce the heading title first to establish context, resulting in a much cleaner and more natural experience for a screen reader user.

Anchor Focus Using JavaScript

In order for multiple in-page anchor links on a webpage to work reliably with screen readers, custom JavaScript must be implemented. Without it, a screen reader can get stuck repeating the link text or focusing on an incorrect location after a jump. This happens because the browser's accessibility focus remains locked onto the clicked link. When the screen reader tries to continue reading from that original position, it conflicts with the visual scroll, causing the viewport to snap back or get stuck at an unpredictable location.

My Implementation:

To fix this, a tabindex of -1 is programmatically applied to the target element (like an article or heading). This opens up the static element to receive focus, allowing us to utilize JavaScript's .focus() method to physically move the screen reader's focus directly to that section of the page.

Note: Single "skip to main content" links at the very top of a page often handle the native anchor focus with no issues. However, when navigating between multiple anchor targets across the same page, JavaScript focus management is required to prevent screen reader confusion.

Page Navigation Focus

My Implementation:

I noticed that sometimes when I would click on one of the links on my website and it opened a new page, the screen reader would get stuck repeating the link information from the previous page, or begin reading from a random element on the page. To fix this issue, I utilized JavaScript to set the focus to the <h1> element of the page when loaded. A slight delay needed to be added; otherwise, the JavaScript focus method would trigger before the screen reader had fully loaded the new DOM content.

Screen Readers and Abbreviations

Using the <abbr> element is the appropriate way to help screen readers properly pronounce abbreviations or acronyms. Sadly, some screen readers, such as VoiceOver, ignore this element and will pronounce the abbreviation instead of the full word or phrase. For example, VoiceOver would typically read A11y as "A eleven y" or "A one one y" instead of "Accessibility".

My Implementation:

The workaround to fix this issue is to utilize visually hidden text. By wrapping the abbreviation in a span tag with aria-hidden="true", we can hide it from the screen reader entirely. Right next to it, we can place a second span tag containing the full word wrapped in a visually-hidden class. This makes it so sighted users see the abbreviation, while the screen reader bypasses it completely to read the full word instead.

Opening Links

Opening Links in New Tabs (or Not)
My Implementation:

I have always created links on my webpages using target="_blank" to open them in new tabs. For most screen readers though, this causes a problem and complexity for the user. To avoid this, I have left links to open in the same tab by default. I believe this issue is tied to WCAG 3.2.5, where users should have full control of major content changes.
