Target Sizing by Changing Display Type

Providing large, predictable touch or click areas ensures that users can easily activate links without missing them. While text links are traditionally inline, changing their box layout structure alters both visual layout and interactive behavior.

My Implementation:

Instead of leaving my directory links in the sidebar as inline elements, I used CSS to change their box structure to display: block (or display: inline-block). By doing this, the links now have a larger interactive area because they expand to the full width of the sidebar, rather than only around the small bit of text. This helps support WCAG Success Criterion 2.5.5 (Target Size), making the links much easier to activate while also fixing multi-line text wrapping alignments (wrapped text no longer has odd indents).

Avoiding Redundancy (Sidebar Landmark)

Using semantic HTML like <aside> with a <nav> wrapped inside it helps create a nice sidebar layout, but nesting these elements can create redundant chatter for screen reader users. When an <aside> holds only a table of contents, announcing both "complementary" (the default announcement for <aside>) and "navigation" adds unnecessary verbosity without providing additional context.

My Implementation:

When building my sidebar directory, I applied role="none" to the outer <aside> element. This removes the default "complementary" landmark announcement while still keeping the HTML structure intact. Screen readers can then focus directly on the nested <nav> element, which is labeled using aria-labelledby to match the <h2> living within it. This keeps the screen reader landmark cleaner and direct (e.g., "Notes Directory, navigation").

Navigational Headings & Hierarchy

From my learning and the WCAG guidelines, it is recommended that heading sequences start with an <h1> and move downward (<h2>, <h3>). However, modern web layouts often place header menus or sidebars above or beside the <main> content. While placing an <h2> in a sidebar before the primary <h1> seems like a hierarchy violation on paper, screen reader users rely on these navigational headings as fast jump targets, making this a practical and widely accepted pattern. It remains best practice to use only one <h1> per page for the main topic, while using <h2> tags for major sub-sections.

My Implementation:

For my header and footer main menu navigations, I removed the hidden <h2> elements and now only use the ARIA attribute like aria-label="Main Menu". On pages containing a sidebar directory, I intentionally kept a visible <h2> heading for both sighted and screen reader users. Although this <h2> appears in the source order before my page's <h1>, it provides an essential heading target without disrupting the overall document flow.

Labeling Navigation Regions Without Hidden Headings

Using visually hidden <h2> elements was once a common way to provide names for <nav> elements. Modern <nav> elements natively support ARIA labeling attributes, allowing us to bypass the need for hidden heading elements entirely. While hidden headings can still occasionally be helpful, they are often unnecessary because <nav> elements are easily navigated as standalone landmark regions.

My Implementation:

Instead of adding extra code to visually hide <h2> elements inside my site navigation, I use the aria-label attribute directly on the <nav> element (e.g., <nav aria-label="Main Menu"> ). This provides a concise announcement from the screen reader, removes redundant repetition, and still gives users all the context they need to navigate the site.

Writing Alt Text

Writing Effective Image Descriptions How to Write Good Alt Text - YouTube Video

Alt text is an essential way to describe non-text content, such as images, to screen reader users. Context is extremely important and should help with how you go about describing a graphic element. It's also good to try and keep descriptions short (1 to 2 sentences) so as to not overwhelm the reader.

Three common image types:
  1. Informative: The page loses meaning if the image is removed. Provide a concise, context-aware description.
  2. Functional: The image acts as a clickable link or button. Describe the destination or action, not the visual appearance of the icon.
  3. Decorative: The image is a background graphic or visual spacer that provides no unique content. Use a null attribute (alt="") to tell the screen reader to silently skip over it; otherwise, the screen reader may default to reading the raw file name aloud.

Bad Phrases: Do not use the phrases "image of" or "picture of." Screen readers automatically announce the element as a graphic.

Proper Grammar: Always use correct grammar and end sentences with a period. This creates natural pauses for screen readers. Additionally, if an image fails to load, sighted users will be able to read a clean text description instead.

Compliance: This practice is tied directly to WCAG Success Criterion 1.1.1 (Non-text Content), which mandates that all non-text content must have a text alternative that serves an equivalent purpose.

Visually Hidden Elements

Visually Hidden CSS Techniques

Landmark regions can be labeled with ARIA attributes, but there are certain accessibility features we sometimes want to include, such as a "Skip to main Content" link, which requires special CSS to hide visually. These features are special because they are hidden from sighted users, but provide screen reader users additional context and sometimes functionality to make the user experience more efficient.

My Implementation:

I created a CSS class that uses modern techniques such as clip-path and shrinking elements down into a 1px box. Using display: none or visibility: hidden won't work here, as those properties remove elements from the page entirely, hiding them from screen readers too.

Anchor Focus Using JavaScript

In order for multiple in-page anchor links on a webpage to work reliably with screen readers, custom JavaScript must be implemented. Without it, a screen reader can get stuck repeating the link text or focusing on an incorrect location after a jump. This happens because the browser's accessibility focus remains locked onto the clicked link. When the screen reader tries to continue reading from that original position, it conflicts with the visual scroll, causing the viewport to snap back or get stuck at an unpredictable location.

My Implementation:

To fix this, a tabindex of -1 is programmatically applied to the target element (like an article or heading). This opens up the static element to receive focus, allowing us to utilize JavaScript's .focus() method to physically move the screen reader's focus directly to that section of the page.

Note: Single "skip to main content" links at the very top of a page often handle the native anchor focus with no issues. However, when navigating between multiple anchor targets across the same page, JavaScript focus management is required to prevent screen reader confusion.

Page Navigation Focus

My Implementation:

I noticed that sometimes when I would click on one of the links on my website and it opened a new page, the screen reader would get stuck repeating the link information from the previous page, or begin reading from a random element on the page. To fix this issue, I utilized JavaScript to set the focus to the <h1> element of the page when loaded. A slight delay needed to be added; otherwise, the JavaScript focus method would trigger before the screen reader had fully loaded the new DOM content.

Screen Readers and Abbreviations

Using the <abbr> element is the appropriate way to help screen readers properly pronounce abbreviations or acronyms. Sadly, some screen readers, such as VoiceOver, ignore this element and will pronounce the abbreviation instead of the full word or phrase. For example, VoiceOver would typically read A11y as "A eleven y" or "A one one y" instead of "Accessibility".

My Implementation:

The workaround to fix this issue is to utilize visually hidden text. By wrapping the abbreviation in a span tag with aria-hidden="true", we can hide it from the screen reader entirely. Right next to it, we can place a second span tag containing the full word wrapped in a visually-hidden class. This makes it so sighted users see the abbreviation, while the screen reader bypasses it completely to read the full word instead.

Opening Links

Opening Links in New Tabs (or Not)
My Implementation:

I have always created links on my webpages using target="_blank" to open them in new tabs. For most screen readers though, this causes a problem and complexity for the user. To avoid this, I have left links to open in the same tab by default. I believe this issue is tied to WCAG 3.2.5, where users should have full control of major content changes.
